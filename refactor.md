# Refactor Plan
_Generated by Compliance Agent — 2026-06-25_

## Summary

The codebase is in generally good shape. The service layer, store architecture, TypeScript strictness, router lazy-loading, and composable structure all follow the guidelines well. However, 31 specific violations were identified across five rule areas: (1) direct Supabase calls inside components and a store — the single most pervasive breach of the service-layer rule; (2) components that exceed the 250-line threshold; (3) `any` type usage in a handful of files; (4) `v-for` index keys on static-content lists; and (5) emit event names that use the imperative form instead of the required past-tense form. Remediation effort is moderate. The critical and high-priority items are concentrated in four page-level components and the `songcode` store.

## Critical
> Violations that break functionality, introduce security risks, or are explicitly forbidden by the guidelines. Must be fixed before the next release.

### Direct Supabase calls inside ArtistsPage component
- **File:** `src/pages/authenticated/ArtistsPage.vue` (lines 148–164)
- **Issue:** Three raw `supabase.from(...)` calls appear inside a `confirmCallback` closure inside the component. The component queries `song_artists_v2`, `library_songs`, and then deletes from `library_songs` directly, bypassing the service layer entirely.
- **Guideline:** "No raw Supabase calls inside components. Components and composables call service functions — never the Supabase client directly." (`API & Data Fetching — Service layer`)
- **Suggested fix:** Extract the three queries plus the delete into a new function (e.g., `bulkDeleteArtistSongs`) in `src/services/artistService.ts`, and call that function from the component.

### Direct Supabase import with no active call in TagsPage
- **File:** `src/pages/authenticated/TagsPage.vue` (line 86)
- **Issue:** `supabase` is imported directly into the component. No `.from()` call was found in the file, meaning this is either dead code from a previous violation or a latent pattern. Either way, components must not import the Supabase client.
- **Guideline:** "No raw Supabase calls inside components." (`API & Data Fetching — Service layer`)
- **Suggested fix:** Remove the `import { supabase }` line. Verify any functionality that previously used it is fully covered by the store or a service.

### Direct Supabase calls inside songcode Pinia store
- **File:** `src/stores/songcode.ts` (lines 25–29, 66–70, 120–125)
- **Issue:** The store makes three separate `supabase.from('songcode')` queries (fetch, upsert for update, upsert for generate JSON). Stores are not the service layer; they should delegate data access to service functions in `src/services/`.
- **Guideline:** "Supabase queries live in `src/services/`. Components and composables call service functions — never the Supabase client directly." (`API & Data Fetching — Service layer`)
- **Suggested fix:** Create `src/services/songcodeService.ts` with `fetchSongcode(songId)`, `upsertSongcode(payload)`, etc. Have `songcode.ts` call those functions.

### Direct Supabase import with no active call in ListsPage
- **File:** `src/pages/authenticated/ListsPage.vue` (line 172)
- **Issue:** `supabase` is imported directly into the page component but no `.from()` call exists in the file body. Same dead-import violation as TagsPage.
- **Guideline:** `API & Data Fetching — Service layer`
- **Suggested fix:** Remove the `import { supabase }` line.

## Medium
> Style and convention issues, naming inconsistencies, or patterns that accumulate technical debt over time.

### Emit event names use imperative rather than past-tense form
- **Files:** `src/components/FilterByTagsModal.vue`, `BulkAddToListsModal.vue`, `BulkAssignTagsModal.vue`, `BulkRemoveTagsModal.vue`, `ManageTagsModal.vue`, `ManageListsModal.vue`, `CRUDModal.vue`, `CRUDEmptyState.vue`, `SongCodeDrawer.vue`, `NoteCard.vue`, `ListTitleCard.vue`, `Card.vue`, `ProjectContactBanner.vue`
- **Issue:** Events named `close`, `submit`, `apply`, `create`, `edit`, `delete`, `open` are imperative commands rather than past-tense descriptions of what happened.
- **Guideline:** "Use descriptive, past-tense event names: `user-selected`, `form-submitted`, `modal-closed`." (`Components — Emits`)
- **Suggested fix:** Rename across all affected files: `close` → `closed`, `submit` → `submitted`, `apply` → `applied`, `create` → `created`, `edit` → `edited`, `delete` → `deleted`, `open` → `opened`. Update all parent-side listeners at the same time.

### `v-for` index keys on iterated lists
- **Files:** `src/components/PatternDisplay.vue` (lines 4, 13, 22), `src/components/SongcodeLyricsContent.vue` (line 9), `src/components/DropdownMenu.vue` (line 26)
- **Issue:** All five `v-for` loops use the loop index as the `:key`. For `DropdownMenu`, items can change dynamically (variant-based separators are inserted between them), making index keys more problematic there.
- **Guideline:** "Always use stable, unique keys on `v-for`. Never use the loop index as a key when the list can be reordered or filtered." (`Performance — v-for keys`)
- **Suggested fix:** For `PatternDisplay` segments, add a stable `id` or key field to the `Seg` interface. For `SongcodeLyricsContent` sections, key on `section.name`. For `DropdownMenu`, key on `item.label`.

### `select('*')` used where only specific columns are consumed
- **Files:** `src/services/artistService.ts` (lines 27, 84), `src/services/tagService.ts` (line 7), `src/services/profileService.ts` (line 7), `src/services/membershipService.ts` (lines 81, 98), `src/services/projectService.ts` (lines 25, 36, 48), `src/stores/songcode.ts` (line 27)
- **Issue:** `.select('*')` is used in at least nine places where only a subset of columns is consumed. This over-fetches data on every call.
- **Guideline:** "Select only the columns you need." (`API & Data Fetching — Avoid over-fetching`)
- **Suggested fix:** Audit what fields are actually used after each query and replace `select('*')` with an explicit column list.

### Types are all in one flat file rather than domain-organized files
- **File:** `src/types/database.ts`
- **Issue:** All domain types (User, Profile, Project, Song, Tag, List, Note, etc.) live in a single file. The guidelines prescribe organizing them by domain.
- **Guideline:** "Domain types live in `src/types/`. Keep them colocated with their domain: `project.ts`, `user.ts`, `common.ts`." (`TypeScript — Shared types`)
- **Suggested fix:** Split `database.ts` into at minimum `project.ts`, `user.ts`, `song.ts`, `note.ts`, `list.ts`, `tag.ts`. Update all import sites. This is a mechanical refactor best done in one focused PR.

### `console.error` / `console.warn` used directly instead of the logger utility
- **Files:** `src/components/ArtistSelector.vue` (lines 152, 181), `src/pages/authenticated/ListDetailPage.vue` (lines 321, 336, 355, 468), `src/components/SongSearchModal.vue` (lines 209, 230, 259), `src/stores/auth.ts` (line 75), `src/stores/songs.ts` (lines 177, 187, 197)
- **Issue:** Raw `console.error` and `console.warn` calls bypass the centralized `logger` utility. Several call sites in components also silently swallow the error without showing a user-facing toast.
- **Guideline:** "Never silently swallow errors — at minimum, log them; ideally surface them to the user via a toast or error state." (`API & Data Fetching — Error handling`)
- **Suggested fix:** Replace raw `console.*` with `logger.error` / `logger.warn` from `@/utils/logger`. For user-facing errors in `SongSearchModal` and `ArtistSelector`, also call `uiStore.showErrorToast(...)`.

### `any` casts in logger.ts and performance.ts
- **Files:** `src/utils/logger.ts` (lines 40, 51, 71), `src/utils/performance.ts` (lines 145, 151, 160)
- **Issue:** `...args: any[]` in logger methods and `as any` / `as any[]` for PerformanceObserver entries, where `PerformanceEntry` subtypes are available from the browser type lib.
- **Guideline:** `TypeScript — Avoid any`
- **Suggested fix:** In `logger.ts`, change variadic args to `...args: unknown[]` and `error?: Error | unknown`. In `performance.ts`, replace `as any[]` with `as PerformanceEntry[]` (or the specific subtype: `LargestContentfulPaint`, `PerformanceEventTiming`, `LayoutShift`).

### `as any` in SongcodeLyricsContent due to incomplete LivenotesJson type
- **File:** `src/components/SongcodeLyricsContent.vue` (lines 110, 114)
- **Issue:** `props.livenotesJson as any` and `(section as any).pattern` are used because the `LivenotesJson` type does not model the `patterns` map or the `pattern` field on sections.
- **Guideline:** `TypeScript — Avoid any`
- **Suggested fix:** Extend the `LivenotesJson` type in `src/types/database.ts` to include the `patterns` record and the `pattern` reference on `Section`, then remove both casts.

## Low

### Generic UI components lack the `Base` prefix
- **Files:** `src/components/Card.vue`, `src/components/LoadingSpinner.vue`, `src/components/DropdownMenu.vue`, `src/components/StickyBar.vue`
- **Issue:** These components contain no business logic and are used across multiple domains. They meet the definition of "base/generic UI components" and should be prefixed with `Base`.
- **Guideline:** "Base/generic UI components (no business logic) are prefixed with `Base`: `BaseButton.vue`, `BaseModal.vue`, `BaseInput.vue`." (`Components — Naming`)
- **Suggested fix:** Rename to `BaseCard.vue`, `BaseLoadingSpinner.vue`, `BaseDropdownMenu.vue`, `BaseStickyBar.vue` and update all import and template-usage sites.

### `supabase.rpc` cast to `any` in service files
- **Files:** `src/services/membershipService.ts` (line 122), `src/services/notePushService.ts` (line 7)
- **Issue:** `(supabase.rpc as any)(...)` is used because the generated Supabase types do not include the RPC functions. This suppresses type safety for RPC calls.
- **Guideline:** `TypeScript — Avoid any`; `TypeScript — Supabase types`
- **Suggested fix:** Regenerate `src/types/supabase.ts` after ensuring the RPC functions are present in the Supabase schema. If they are intentionally absent, use a narrower return-type cast rather than `as any`.

### `as unknown as` double-cast pattern used to bridge type mismatches
- **Files:** `src/stores/songcode.ts` (lines 35, 74, 128), `src/services/projectService.ts` (lines 31, 40, 52), `src/services/membershipService.ts` (lines 84, 117)
- **Issue:** The `data as unknown as SomeType` pattern signals that the Supabase-generated types and the hand-written domain types are not aligned. Each occurrence is a silent type lie.
- **Guideline:** `TypeScript — Supabase types` ("Generate and use Supabase's typed client.")
- **Suggested fix:** Align the hand-written types in `src/types/database.ts` with the generated schema in `src/types/supabase.ts`, eliminating the need for double-casts. As a short-term measure, add a comment at each cast site explaining the mismatch.

### `AnyQuery` alias for `any` in publicLibraryService and settingsService
- **Files:** `src/services/publicLibraryService.ts` (lines 1–10), `src/services/settingsService.ts` (lines 3–4)
- **Issue:** Both files define `type AnyQuery = any` at the top to suppress type errors for tables not yet in the generated schema. The alias name obscures intent.
- **Guideline:** `TypeScript — Avoid any`
- **Suggested fix:** Once the `public_libraries` migration is applied and types are regenerated, remove the alias. Until then, add an explanatory `// TODO: remove after schema migration` comment.

## Compliant areas

- **Router (`src/router/index.ts`):** All routes use dynamic `() => import(...)` — full compliance with the lazy-loading guideline.
- **Service layer (most files):** `artistService`, `noteService`, `listService`, `libraryService`, `listItemService`, `tagService`, `songService`, `authService`, `profileService`, `membershipService`, and `publicLibraryService` all correctly centralize Supabase access with proper error propagation.
- **Composables (`src/composables/`):** `useCRUD`, `useArtistFormList`, `useOnlineStatus`, `useOfflineSync`, and `usePageLoad` all follow the naming convention, live in the correct directory, avoid component coupling, and return only what consumers need.
- **TypeScript prop and emit declarations:** `defineProps<{...}>()` and `defineEmits<{...}>()` are used in typed generic form throughout all components. No runtime-style object-argument variants were found.
- **Pinia stores (except songcode):** All stores correctly delegate data fetching to service functions and do not import the Supabase client.
- **Error handling in services:** All service functions consistently check both `data` and `error` from Supabase and throw on error rather than silently continuing.
- **Component naming (PascalCase and domain-prefix):** All `.vue` files use PascalCase. Domain-prefix naming (`NoteCard`, `ProjectSettingsDrawer`, `ArtistSelector`, etc.) is consistently applied.
- **`withDefaults` for optional props:** Used correctly wherever props have sensible defaults (`LoadingSpinner`, `PatternDisplay`, `SongcodeLyricsContent`).
