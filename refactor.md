# Refactor Plan
_Generated by Compliance Agent — 2026-05-01_

## Summary

The codebase is in a substantially better state than it was at the last audit. Several previously flagged issues have been resolved: `src/types/supabase.ts` now exists as a proper generated file; the Supabase client correctly lives at `src/lib/supabase.ts`; `useArtistFormList` has been extracted and is shared by both song pages; `useNotesDrawer` is shared between `LibraryPage` and `ListDetailPage`; `ListDetailPage` now imports from `src/services/` rather than calling Supabase directly; and `src/services/libraryService.ts` and `src/services/listItemService.ts` exist with real logic. The component size limit violations that were previously flagged are also resolved — `LibraryPage.vue` is now 569 lines (still over limit) and `ListDetailPage.vue` is now 551 lines (still over limit), but both are substantially smaller than before.

What remains is 17 violations across all four severity levels. The most pressing issue is that raw Supabase calls still dominate every Pinia store — none of the store queries have been extracted to service functions. The `any` type is pervasive across stores and the sole service file. Several `select('*')` over-fetches remain. `console.error` and `console.warn` calls are scattered throughout stores and pages. The two page components still exceed 250 lines. Remediation effort is moderate: the service layer extraction is the largest task but can be done domain by domain without breaking behaviour.

### `any` casts in the join-result transformation pattern across stores and services

- **Files:** `src/stores/library.ts` (lines 123, 124, 128, 130, 132, 282, 283, 287, 289, 291), `src/stores/songs.ts` (lines 77, 79, 82, 116, 118, 121, 361, 385, 411, 414), `src/stores/globalSongs.ts` (lines 51, 53, 56, 138, 140, 143), `src/stores/lists.ts` (lines 87, 90, 92, 93, 151), `src/stores/artists.ts` (lines 71, 73), `src/services/libraryService.ts` (lines 35, 36, 39, 42, 43, 44)
- **Issue:** The join-result transform pattern (`sa: any`, `lst: any`, `li: any`) appears in at least 30 places across 6 files. `cleanUpdates: any` in `lists.ts` line 151 directly bypasses TypeScript on a write operation. `(supabase as any).rpc(...)` in `library.ts` line 170 casts the entire client to `any` to call an RPC that is missing from the generated types.
- **Guideline:** TypeScript — Avoid `any`: "Using `any` is a code smell, not a shortcut." "If you genuinely don't know the type, use `unknown` and narrow it explicitly."
- **Suggested fix:** Define intermediate transformation interfaces in `src/types/database.ts`: `SongArtistJoinRow`, `TagJoinRow`, `ListJoinRow`, using the actual Supabase-generated types from `src/types/supabase.ts`. Replace every `sa: any`, `lst: any`, `li: any` cast. Replace `cleanUpdates: any` with `Partial<Pick<List, 'name' | 'description'>>`. For the missing RPC type, extend the `Database` type in a local augmentation file or add the RPC signature to `src/types/supabase.ts` manually until the schema is regenerated.

### `any` used in `SongNotesDrawer.vue` icon map

- **File:** `src/components/SongNotesDrawer.vue` (line 223)
- **Issue:** `getNoteIcon` returns `Record<string, any>` and the map values are render functions. This loses the specific function signature and means the return type of `getNoteIcon` is `any`.
- **Guideline:** TypeScript — Avoid `any`.
- **Suggested fix:** Type the icon map as `Record<string, () => VNode>` using the `VNode` type from Vue. Import `VNode` from `vue` alongside the existing `h` import.

### `any` used in `SongSearchModal.vue` error catch clause

- **File:** `src/components/SongSearchModal.vue` (line 223)
- **Issue:** `catch (err: any)` is used to access `err?.message` and `err?.code`. The `any` annotation on a catch clause parameter defeats strict TypeScript error handling.
- **Guideline:** TypeScript — Avoid `any`.
- **Suggested fix:** Use `catch (err: unknown)` and narrow with `instanceof Error` for `err.message`. For `err.code`, define a small `SupabaseError` interface with a `code` string field and use a type guard before accessing it.

### `any` used in `ListDetailPage.vue` — `editingTitle`, `handleEditTitle`, `handleOpenNotes`

- **File:** `src/pages/ListDetailPage.vue` (lines 286, 453, 536)
- **Issue:** `editingTitle` is typed as `ref<any>(null)`. `handleEditTitle(item: any)` and `handleOpenNotes` casts its parameter with `(item as any).library_song_id`.
- **Guideline:** TypeScript — Avoid `any`.
- **Suggested fix:** Type `editingTitle` as `ref<ListItem | null>(null)`. `handleEditTitle` should accept `ListItem`. For `handleOpenNotes`, `library_song_id` is already a field on the `ListItem` interface in `src/types/database.ts` — access it directly without a cast.

### `handleDeleteNote` in `useNotesDrawer` does not actually delete

- **File:** `src/composables/useNotesDrawer.ts` (lines 40–52)
- **Issue:** `handleDeleteNote` shows a confirm dialog, then on confirmation shows a stub toast `'Delete note functionality coming soon'` and does nothing. Users who confirm a deletion believe data has been removed; it has not. This is a functional correctness problem masquerading as incomplete work.
- **Guideline:** API & Data Fetching — Error handling: "Never silently swallow errors — at minimum, log them; ideally surface them to the user."
- **Suggested fix:** Implement the actual deletion by calling `useNotesStore().deleteNote(note.id, note.library_song_id)`. If the feature is genuinely not ready, remove the confirm prompt entirely and disable or hide the delete button until it is implemented, so the user is never given a false confirmation.

### `console.error` and `console.warn` calls left in production code

- **Files:** `src/stores/auth.ts` (lines 41, 84), `src/stores/songs.ts` (lines 363, 388, 416), `src/stores/settings.ts` (lines 42, 57, 96, 122, 135), `src/composables/useNotesDrawer.ts` (line 27), `src/pages/ListDetailPage.vue` (lines 366, 381, 479, 499, 547), `src/lib/supabase.ts` (implicit — throws on missing env vars, which is correct)
- **Issue:** `console.error` and `console.warn` calls are scattered across stores, a composable, and a page. These leak internal error details to the browser console in production. The project already uses `src/utils/logger.ts` (imported in `usePageLoad`), which is the designated logging utility.
- **Guideline:** Core Principles — DRY / clean code. The guidelines specify a "clean, consistent" codebase; `logger` already exists for this purpose.
- **Suggested fix:** Replace `console.error(...)` and `console.warn(...)` calls with `logger.error(...)` and `logger.warn(...)` from `src/utils/logger.ts`. In `src/stores/songs.ts`, the three `refreshSong*` helpers swallow errors silently after logging — consider re-throwing or surfacing them via the store's `error` state.

### `select('*')` over-fetching in stores and service

- **Files:** `src/stores/artists.ts` (line 28: `.select('*')` on `artists`), `src/stores/tags.ts` (line 26: `.select('*')` on `tags`), `src/stores/library.ts` (line 84: `.select('*,…')` on `library_songs` base), `src/stores/globalSongs.ts` (line 163: `.select('*')` on `artists_v2`), `src/services/libraryService.ts` (line 19: `notes:notes(*)`)
- **Issue:** The guideline requires selecting only the columns required. `artists` and `tags` tables are narrow, but `library_songs` and `artists_v2` carry columns never used in the UI (e.g. `fingerprint`, `merge_reason`, `merged_into_id`, `verified_*`). The `notes:notes(*)` wildcard in `libraryService.ts` fetches `share_token` and `created_by`/`updated_by` fields that the notes list drawer never displays.
- **Guideline:** API & Data Fetching — Avoid over-fetching: "Select only the columns you need."
- **Suggested fix:** For `libraryService.ts` notes embed, replace `notes:notes(*)` with an explicit column list matching what `SongNotesDrawer` actually renders: `notes:notes(id, type, title, content, data, display_order, updated_at, is_public, is_shareable)`. For `artists` and `tags`, the wildcard is acceptable given the narrow schemas, but the `songs_v2` and `artists_v2` selects should be tightened to exclude merge/verify metadata columns not used in the app.

### Duplicated artist-transform logic between `library.ts` and `libraryService.ts`

- **Files:** `src/stores/library.ts` (lines 118–133, 277–292), `src/services/libraryService.ts` (lines 31–45)
- **Issue:** The three-step transform (map artist join rows, filter nulls, sort by position), the tag flatten, and the list flatten are copy-pasted across `loadLibrary`, `getLibrarySongById`, and `fetchLibrarySongWithDetails`. Any change to the join shape requires updates in three places. This is a DRY violation within the same domain.
- **Guideline:** Core Principles — DRY: "Extract repeated logic into a composable" / "Extract repeated Supabase queries into a service function."
- **Suggested fix:** Once the service layer extraction (Critical finding above) is complete, consolidate all three callers into `libraryService.ts` as a single `fetchLibrarySongWithDetails(id)` and a list variant `fetchLibrarySongs(projectId)`. The store's `loadLibrary` and `getLibrarySongById` become thin wrappers calling the service. The transformation helper can be extracted to a private `transformLibrarySongRow` function inside `libraryService.ts`.

### `reorderNotes` issues N individual UPDATE queries in a loop

- **File:** `src/stores/notes.ts` (lines 282–297)
- **Issue:** `reorderNotes` sends one `supabase.from('notes').update(...)` call per note in a `for` loop. For a song with 10 notes this fires 10 sequential round-trips. `reorderListItems` in `lists.ts` correctly uses a single `supabase.rpc('update_list_item_positions', ...)` batch call. The inconsistency suggests the notes version was never updated to match.
- **Guideline:** Performance — implicit from KISS and avoid over-fetching; multiple serial network requests where one would suffice is a performance anti-pattern.
- **Suggested fix:** Either create an equivalent RPC function `update_note_positions` in Supabase and call it the same way `reorderListItems` does, or use a single `upsert` with the full array of `{ id, display_order }` objects. The loop approach should be removed.
