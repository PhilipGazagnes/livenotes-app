# Refactor Plan
_Generated by Compliance Agent ‚Äî 2026-05-01_

## Summary

The codebase is in reasonably good shape structurally. The service layer, composables, and Pinia stores are well separated, and the Supabase typed client is in use. However, the audit found **22 violations** across all four severity levels. The most concentrated problems are in the TypeScript discipline category ‚Äî `any` is used extensively throughout the services and components ‚Äî and in the API/data-fetching layer, where several raw Supabase calls appear directly inside a store and inside a component. Component size is also a recurring concern, with multiple page components well over the 250-line limit. Fixing the critical and high items is the most impactful work; the medium and low items are primarily polish.

## Critical

> Violations that break functionality, introduce security risks, or are explicitly forbidden by the guidelines. Must be fixed before the next release.

### Raw Supabase calls inside `auth` store
- **File:** `src/stores/auth.ts` (lines 24‚Äì82, 101, 109, 140, 168, 196, 216)
- **Issue:** Every Supabase query in this file ‚Äî `supabase.from('projects')`, `supabase.auth.*`, `supabase.from('library_songs')` ‚Äî is executed directly inside the store. The guideline states "No raw Supabase calls inside components" and that all queries must live in `src/services/`. Stores are not listed as an exception; they are consumers of service functions, not owners of query logic.
- **Guideline:** _API & Data Fetching ‚Äî Service layer_: "Components and composables call service functions ‚Äî never the Supabase client directly."
- **Suggested fix:** Extract `createPersonalProject`, `loadPersonalProject`, and the auth calls into `src/services/projectService.ts` and `src/services/authService.ts`. The store should import and call those service functions.

### Raw Supabase call inside `libraryService.ts` using unsafe cast
- **File:** `src/services/libraryService.ts` (line 114)
- **Issue:** `await (supabase as any).rpc('increment_song_popularity', ...)` casts the typed Supabase client to `any` to call an RPC function. This bypasses all type safety and silently drops any error handling (the return value is not checked). The `increment_song_popularity` RPC does not appear in `src/types/supabase.ts`, so either the generated types are stale or the cast is used to hide a type mismatch.
- **Guideline:** _TypeScript ‚Äî Avoid `any`_: "Using `any` is a code smell, not a shortcut." Also, _API & Data Fetching ‚Äî Error handling_: "Never silently swallow errors."
- **Suggested fix:** Add `increment_song_popularity` to the `Functions` section in `src/types/supabase.ts` by regenerating the types (`npx supabase gen types typescript ...`). Then call `supabase.rpc('increment_song_popularity', ...)` with proper `{ data, error }` destructuring and throw on error.

## High

> Significant deviations that affect consistency, maintainability, or correctness. Should be addressed in the current or next sprint.

### Pervasive use of `any` in service layer transform functions
- **File:** `src/services/songService.ts` (lines 18, 21, 22, 24, 26, 57, 58, 61, 63, 70, 147, 157, 167, 170)
- **File:** `src/services/libraryService.ts` (lines 35, 37, 40, 43, 78, 82, 85, 87)
- **File:** `src/services/listService.ts` (lines 54, 55, 56, 57, 63)
- **File:** `src/services/artistService.ts` (lines 37, 39, 40)
- **Issue:** Every data-transform function uses `any` to type the raw rows returned by Supabase joins (`st: any`, `sa: any`, `li: any`, `ls: any`, etc.). This defeats the purpose of the typed Supabase client. The join shapes are predictable and can be typed with inline types or with `Tables` helpers from `src/types/supabase.ts`.
- **Guideline:** _TypeScript ‚Äî Avoid `any`_: "If you genuinely don't know the type, use `unknown` and narrow it explicitly."
- **Suggested fix:** Define local interfaces for the raw join shapes returned by Supabase (e.g., `interface RawSongRow`, `interface RawLibrarySongRow`) or use the generated `Tables<'songs'>` type intersected with nested relation types. Replace all `any` parameters with these typed interfaces.

- **File:** `src/services/songService.ts` (line 70): `{ ...songData, created_by: userId, updated_by: userId } as any`
- **File:** `src/services/noteService.ts` (line 29): `data: payload.data as Record<string, any>`
- **File:** `src/services/noteService.ts` (line 50): `data: updates.data as Record<string, any> | null`
- **Issue:** Insert and update payloads are cast to `any` or to `Record<string, any>` to silence TypeScript, which masks potential type mismatches between the app's domain types and the database schema.
- **Guideline:** _TypeScript ‚Äî Avoid `any`_
- **Suggested fix:** For `songService.ts` line 70, use the generated `TablesInsert<'songs'>` type. For `noteService.ts`, the `data` JSONB column is typed as `Json` in `supabase.ts`; cast `SongcodeNoteData | LooperNoteData` to `Json` explicitly (not through `any`) with a type assertion or a conversion helper.

### `any` used in `ArtistInput.vue` component

### `select('*')` used where only specific columns are needed
- **File:** `src/services/tagService.ts` (line 5): `.select('*')` on `tags` ‚Äî only `id` and `name` are used by the UI.
- **File:** `src/services/artistService.ts` (line 6): `.select('*')` on `artists` ‚Äî only `id`, `name`, and `created_at` are used.
- **File:** `src/services/songService.ts` (line 33): uses the `SONG_SELECT` constant which starts with `*` ‚Äî justified for the full song detail view, but not for bulk list operations.
- **File:** `src/stores/auth.ts` (line 54): `.select('id, created_at')` ‚Äî this is already scoped correctly. But the inner `library_songs` count query (line 73) uses `.select('*', { count: 'exact', head: true })` which is correct idiom for count-only queries.
- **Issue:** Fetching full rows when only a subset of columns is consumed wastes bandwidth and reduces query clarity.
- **Guideline:** _API & Data Fetching ‚Äî Avoid over-fetching_: "Select only the columns you need."
- **Suggested fix:** In `tagService.ts`, change to `.select('id, name, project_id, created_at')`. In `artistService.ts`, change to `.select('id, name, project_id, created_at, updated_at')`. Audit `SONG_SELECT` in `songService.ts` and remove columns not consumed by the UI.

### `useNotesDrawer` composable has unimplemented functionality

- **Issue:** `handleDeleteNote` shows a confirmation dialog but then calls `uiStore.showToast('Delete note functionality coming soon', 'success')` ‚Äî it does not actually delete the note. ## Compliant areas

The following areas are in good shape and required no findings:

- **Service layer existence and structure:** All Supabase table queries are located in `src/services/` (song, note, tag, list, artist, library services) and are imported by stores, not by components directly.
- **Supabase typed client:** `src/lib/supabase.ts` uses `createClient<Database>()` with the generated types from `src/types/supabase.ts`, which is present and current.
- **Composable naming and placement:** All composables (`useNotesDrawer`, `useCRUD`, `usePageLoad`, `useArtistFormList`) are prefixed with `use` and reside in `src/composables/`.
- **`v-for` key discipline:** All audited `v-for` loops use stable entity IDs (`.id` or `slot.id`) as keys, not loop indices.
- **Route-level lazy loading:** All page components in `src/router/index.ts` use dynamic `import()` for lazy loading.
- **`defineAsyncComponent` for heavy components:** `SongSearchModal`, `ManageTagsModal`, and others in `LibraryPage.vue` and `SongEditPage.vue` are correctly lazy-loaded via `defineAsyncComponent`.
- **Props and emits declaration:** The majority of components use `defineProps<{...}>()` and `defineEmits<{...}>()` with TypeScript generics, as required.
- **Error handling in service functions:** Service functions consistently destructure `{ data, error }` from Supabase and throw on error rather than silently swallowing failures.
- **Realtime subscription cleanup:** No Realtime subscriptions were found that are missing `onUnmounted` cleanup (no subscriptions are present in the audited files).
- **`src/types/` structure:** Domain types (`database.ts`, `supabase.ts`) are colocated in `src/types/` as required.
