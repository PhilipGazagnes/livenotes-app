### Direct Supabase client calls inside App.vue

- **File:** `src/App.vue` (lines 4, 17, 19)
- **Issue:** `App.vue` imports the Supabase client directly (`import { supabase } from '@/lib/supabase'`) and calls `supabase.auth.startAutoRefresh()` and `supabase.auth.stopAutoRefresh()` inside the component. The guideline explicitly forbids raw Supabase calls in components — all Supabase interactions must go through `src/services/`.
- **Guideline:** API & Data Fetching — Service layer: "Components and composables call service functions — never the Supabase client directly."
- **Suggested fix:** Add `startAutoRefresh` and `stopAutoRefresh` wrappers to `src/services/authService.ts`, then import and call those wrappers from `App.vue`. Alternatively, extract the entire visibility-change logic into a composable (e.g., `src/composables/useAuthRefresh.ts`) that delegates to `authService`.

### fetchListItemCount silently swallows the Supabase error

- **File:** `src/services/listService.ts` (lines 190–197)
- **Issue:** The Supabase `error` return value is never destructured. If the query fails, the function silently returns `0` rather than propagating the error. Every other function in this same file correctly handles `error`. This is an inconsistency that can hide real database failures.
- **Guideline:** API & Data Fetching — Error handling: "Always handle both the `data` and `error` return values from Supabase. Never silently swallow errors."
- **Suggested fix:**
  ```ts
  export async function fetchListItemCount(listId: string): Promise<number> {
    const { data, error } = await supabase
      .from('list_items')
      .select('id')
      .eq('list_id', listId)
      .eq('type', 'song')
    if (error) throw error
    return data?.length ?? 0
  }
  ```

- **File:** `src/composables/useOfflineSync.ts` (lines 23–24)
- **Issue:** A `localStorage.getItem` call and a mutation of `lastSyncedAt` run at the top level of the composable function body — outside any named function — every time the composable is instantiated. The guideline requires that side effects in composables be contained inside clearly named functions.
- **Guideline:** Composables — Rules: "Avoid side effects outside of clearly named functions."
- **Suggested fix:** Move the `localStorage` read into a named initializer called inside `onMounted`, or perform it lazily at the start of `warmUp`. Example using `onMounted`:
  ```ts
  export function useOfflineSync() {
    const isSyncing = ref(false)
    const progress = ref<SyncProgress | null>(null)
    const lastSyncedAt = ref<Date | null>(null)

    function loadPersistedSyncDate(): void {
      const stored = localStorage.getItem(STORAGE_KEY)
      if (stored) lastSyncedAt.value = new Date(stored)
    }

    onMounted(loadPersistedSyncDate)
    // ...
  }
  ```

### fetchListWithItems uses wildcard select on multiple levels

- **File:** `src/services/listService.ts` (lines 49–67)
- **Issue:** The nested Supabase join query uses bare `*` at the `list_items`, `library_songs`, `songs_v2`, `artists_v2`, `tags`, and `lists` levels. All columns from every joined table are fetched even though only a defined subset (modelled by `RawListItemRow` and its nested interfaces) is actually consumed.
- **Guideline:** API & Data Fetching — Avoid over-fetching: "Select only the columns you need: `.select('id, name, status')` instead of `.select('*')` when the full row isn't required."
- **Suggested fix:** Replace each `*` with an explicit column list that matches what the local raw-row interfaces declare. This reduces payload size and makes the data contract self-documenting.

### useOnlineStatus uses lifecycle hooks, making it unsafe outside component context

- **File:** `src/composables/useOnlineStatus.ts` (lines 10–18)
- **Issue:** The composable registers `window` event listeners via `onMounted` and cleans them up via `onUnmounted`. This lifecycle coupling means the composable only works correctly when called inside a component `<script setup>`. Using it in a store, another composable, or a service context (as could happen as the offline-sync feature grows) will silently skip listener registration with no warning.
- **Guideline:** Composables — Rules: "Don't couple composables to specific components — keep them generic and reusable."
- **Suggested fix:** Use `tryOnMounted` / `tryOnUnmounted` from `@vueuse/core` (already a project dependency) so the composable degrades gracefully outside a component context. Alternatively, since `@vueuse/core` is present, replace the implementation entirely with the built-in `useOnline()` composable, which uses a singleton event listener at module scope.
