# Refactor Plan
_Generated by Compliance Agent — 2026-05-01_

## Summary

The codebase is in a functional but structurally non-compliant state across several key dimensions defined in `./ai/dev-guidelines.md`. A total of 22 distinct violations were found spanning all four severity levels. The most critical area is the complete absence of a service layer: every Supabase query lives directly in Pinia stores or, worse, inside Vue components. Two pages (`LibraryPage.vue` at 661 lines, `ListDetailPage.vue` at 733 lines) substantially exceed the 250-line component size limit. Duplicated logic between `SongNewPage.vue` and `SongEditPage.vue` is the most visible DRY violation. `any` types appear in at least 11 files. Remediation effort is moderate-to-high: the service layer extraction alone will touch every store, but the patterns are consistent enough that it can be done incrementally by domain without breaking existing behaviour.

## Critical
> Violations that break functionality, introduce security risks, or are explicitly
> forbidden by the guidelines. Must be fixed before the next release.

### No service layer — raw Supabase calls inside components
- **File:** `src/pages/LibraryPage.vue` (lines 574–632), `src/pages/ListDetailPage.vue` (lines 456–515, 556–675)
- **Issue:** Both page components call `supabase` directly, executing multi-table join queries inside `handleNoteSaved`, `handleSaveTitle`, `handleDeleteTitle`, and `handleOpenNotes`. This is the most egregious breach of the guideline that states "No raw Supabase calls inside components."
- **Guideline:** API & Data Fetching — Service layer: "Components and composables call service functions — never the Supabase client directly."
- **Suggested fix:** Create `src/services/libraryService.ts` and move these queries there. `handleNoteSaved` / `handleOpenNotes` perform identical `library_songs` refresh queries in both files — a single `fetchLibrarySongById(id)` service function would eliminate both violations and the duplication simultaneously.

### No `src/services/` directory exists
- **File:** entire `src/` tree
- **Issue:** The guideline mandates a `src/services/` directory with domain-scoped files (`projectService.ts`, `userService.ts`, etc.). No such directory or files exist. All Supabase logic lives in Pinia stores, which conflates data-fetching with state management.
- **Guideline:** API & Data Fetching — Service layer: "Supabase queries live in `src/services/`, organized by domain."
- **Suggested fix:** Create the directory and begin extracting queries domain by domain: `libraryService.ts`, `songService.ts`, `listService.ts`, `tagService.ts`, `artistService.ts`, `noteService.ts`. Stores should import and call service functions rather than calling `supabase` directly.

### Supabase generated types file missing
- **File:** `src/types/supabase.ts` — does not exist
- **Issue:** The guideline requires a generated Supabase types file kept at `src/types/supabase.ts`. Its absence means all database interactions are typed by hand, which is error-prone and diverges from the schema silently.
- **Guideline:** TypeScript — Supabase types: "Generate and use Supabase's typed client. Keep the generated types file at `src/types/supabase.ts`."
- **Suggested fix:** Run `npx supabase gen types typescript --project-id YOUR_PROJECT_ID > src/types/supabase.ts` and wire the typed client into the Supabase instantiation in `src/utils/supabase.ts`.

- **File:** `src/pages/LibraryPage.vue` (661 lines)

- **Guideline:** Components — Component Size: "If a component exceeds ~250 lines (template + script combined), treat it as a signal to refactor."

### Duplicated artist management logic — SongNewPage and SongEditPage
- **File:** `src/pages/SongNewPage.vue` (lines 239–265), `src/pages/SongEditPage.vue` (lines 262–288)
- **Issue:** The four functions `handleAddArtist`, `handleRemoveArtist`, `handleMoveArtistUp`, and `handleMoveArtistDown` are copied verbatim between the two pages. This is a textbook DRY violation: a future change to artist ordering behaviour must be made in two places.
- **Guideline:** Core Principles — DRY: "Extract repeated logic into a composable."
- **Suggested fix:** Create `src/composables/useArtistFormList.ts` that accepts and returns the reactive `artistIds` array along with the four handler functions. Both pages then call `const { artistIds, handleAddArtist, ... } = useArtistFormList()`.

### Duplicated library-song refresh logic — LibraryPage and ListDetailPage
- **File:** `src/pages/LibraryPage.vue` (lines 571–633), `src/pages/ListDetailPage.vue` (lines 613–676)
- **Issue:** `handleNoteSaved` performs an identical multi-join `library_songs` select and transforms the result the same way in both files. Even the `any` casts and `.filter(Boolean)` calls match line for line.
- **Guideline:** Core Principles — DRY: "Extract repeated Supabase queries into a service function."
- **Suggested fix:** Move the fetch-and-transform logic into a `fetchLibrarySongById(id)` function in `src/services/libraryService.ts`. Both pages call it and receive a typed `LibrarySongWithDetails`.

### Dual `onMounted` hooks causing double data fetch — ListsPage.vue
- **File:** `src/pages/ListsPage.vue` (lines 266–279 and 298–306)
- **Issue:** There are two separate `onMounted` hooks. The first (line 266) calls `fetchLists` manually with its own try/catch/finally. The second (line 298) calls `fetchLists` again through `usePageLoad`. This causes two network requests on every mount and produces a race condition where the overlay is hidden by the first hook before the second has completed.
- **Guideline:** Core Principles — DRY; Components — Single Responsibility.
- **Suggested fix:** Remove the first `onMounted` block entirely (lines 266–279). The `usePageLoad` version on line 298 already handles auth initialization, timeout protection, error toasting, and overlay cleanup.

### Supabase client not at the canonical path
- **File:** `src/utils/supabase.ts`
- **Issue:** The guideline specifies the client should live at `src/lib/supabase.ts`. The actual location is `src/utils/supabase.ts`. All stores import from the non-canonical path.
- **Guideline:** API & Data Fetching — Service layer example: `import { supabase } from '@/lib/supabase'`.
- **Suggested fix:** Move the file to `src/lib/supabase.ts` and update all imports. This is a rename, not a logic change.

## Medium

### `any` used in shared type definitions
- **File:** `src/types/database.ts` (lines 92, 98, 220), `src/stores/notes.ts` (line 97)
- **Issue:** `livenotes_json: any | null`, `external_links: any | null`, and the `data` union on `Note` use `any`. The `createNote` service parameter also accepts `Record<string, any>`. These are in the most-shared parts of the codebase and propagate `any` into every consumer.
- **Guideline:** TypeScript — Avoid `any`: "If you genuinely don't know the type, use `unknown` and narrow it explicitly."
- **Suggested fix:** Replace `any` with `unknown` on `livenotes_json` and `external_links`. For `Note.data`, the union `SongcodeNoteData | LooperNoteData | Record<string, unknown> | null` is already partially defined; tighten it to `SongcodeNoteData | LooperNoteData | null` and add a narrowing guard where the generic case is needed.

### `any` casts in data transformation chains across stores
- **File:** `src/stores/library.ts` (lines 123–133), `src/stores/songs.ts` (lines 77–83, 114–119), `src/stores/globalSongs.ts` (lines 51–56, 138–143), `src/stores/lists.ts` (line 151), `src/stores/artists.ts` (lines 71–88)
- **Issue:** The artist/tag/list join-result transformation pattern (`sa: any`, `lst: any`, `li: any`) is repeated extensively. The `cleanUpdates: any` in `lists.ts` `updateList` is a direct bypass of TypeScript.
- **Guideline:** TypeScript — Avoid `any`.
- **Suggested fix:** Once `src/types/supabase.ts` is generated, replace ad-hoc casts with Supabase-generated join types. In the meantime, define intermediate transformation interfaces (e.g., `SongArtistRow`, `TagJoinRow`) in `src/types/database.ts` to replace `any` casts. Replace `cleanUpdates: any` with a typed `Partial<Pick<List, 'name' | 'description'>>`.

### `any` used in component internals
- **File:** `src/components/SongNotesDrawer.vue` (lines 156, 167, 233–268), `src/components/SongSearchModal.vue` (line 223)
- **Issue:** `currentLyricsJson` is typed as `ref<any | null>`, `getLivenotesJson` returns `any | null`, and `getNoteIcon` maps to `Record<string, any>`. In `SongSearchModal`, the error is caught as `err: any` to access `err.message` and `err.code`.
- **Guideline:** TypeScript — Avoid `any`.
- **Suggested fix:** Type `currentLyricsJson` and `getLivenotesJson` with the actual `livenotes_json` shape (or `unknown` with a type guard). Replace `Record<string, any>` icon map with `Record<string, () => VNode>`. In `SongSearchModal`, use `catch (err: unknown)` and narrow with `instanceof Error` and an interface for the Supabase error code.

### `v-for` loop index used as key on reorderable lists
- **File:** `src/pages/SongNewPage.vue` (line 43), `src/pages/SongEditPage.vue` (line 54)
- **Issue:** Both pages use `:key="index"` on the artist input list. The guideline explicitly prohibits index keys when the list can be reordered or filtered. The move-up/move-down buttons confirm the list can be reordered.
- **Guideline:** Performance — `v-for` keys: "Never use the loop index as a key when the list can be reordered or filtered."
- **Suggested fix:** Give each artist slot a stable identity. The simplest approach: change `artistIds` from `(string | null)[]` to `{ id: string; artistId: string | null }[]` where `id` is a locally-generated UUID (`crypto.randomUUID()`). Use `:key="slot.id"`.

### Leftover `console.log` debug statements in production stores and components
- **File:** `src/stores/ui.ts` (lines 188–189, 205–211, 226, 227), `src/components/SongNotesDrawer.vue` (lines 189–196), `src/components/LibrarySongCard.vue` (lines 144–148), `src/pages/ListDetailPage.vue` (lines 356, 700)
- **Issue:** Multiple `console.log` calls with debug emoji prefixes (`🎼`, `🎵`, `📝`, `✏️`, `📋`, `🎸`) are left in production code across a store and three components. These leak internal data shapes to the browser console.
- **Guideline:** Implicit from DRY / clean code principle; also a standard production quality expectation noted in the guidelines' "clean, consistent" phrasing.
- **Suggested fix:** Remove all `console.log` calls that are not error-level. If diagnostic logging is needed in development, gate it behind `if (import.meta.env.DEV)` or wire it through the existing `src/utils/logger.ts` utility already used in `usePageLoad`.

### `select('*')` over-fetching in stores
- **File:** `src/stores/library.ts` (line 82: `.select('*'...` on `library_songs`), `src/stores/notes.ts` (line 73: `.select('*')` on `notes`), `src/stores/lists.ts` (line 26: `.select('*')` on `lists`, line 48: `.select('*')` on `lists`)
- **Issue:** The guideline recommends selecting only the columns required rather than using `select('*')`. While the `library_songs` select does expand to explicit columns for nested joins, the base table wildcard still fetches unused columns (e.g., `added_by`, `custom_notes`). The `notes` and `lists` wildcards are the cleanest targets.
- **Guideline:** API & Data Fetching — Avoid over-fetching: "Select only the columns you need."
- **Suggested fix:** Replace `.select('*')` for `notes` with `.select('id, library_song_id, type, title, content, data, display_order, created_at, updated_at, is_public, is_shareable')`. Replace `lists` selects with `.select('id, project_id, name, description, created_at, updated_at, created_by')`.

### Composable `useToast` adds no value over direct store access
- **File:** `src/composables/useToast.ts`
- **Issue:** The composable is a three-line pass-through that simply re-exports two methods from `useUiStore`. It is consumed in only one component (`SongSearchModal.vue`) while all other components call `useUiStore` directly. This inconsistency adds confusion without benefit.
- **Guideline:** Composables — When to create a composable: "Logic is used in more than one component / Logic involves reactive state that belongs outside the template." A one-liner wrapper around a single store is neither.
- **Suggested fix:** Remove `useToast.ts` and replace the single usage in `SongSearchModal.vue` with `const uiStore = useUiStore()` directly, consistent with the rest of the codebase.

## Compliant areas

The following areas are well-structured and consistent with the guidelines:

- **Router (`src/router/index.ts`):** Route-level lazy loading is applied to all pages via `() => import(...)`, as required. The auth guard is clean and correctly uses the store.
- **`usePageLoad` composable (`src/composables/usePageLoad.ts`):** Well-typed, focused, with clear `loading`/`error` state exposure and a documented interface. A good example of what service-layer composables should look like.

- **`LibrarySongCard.vue`, `TagCard.vue`, `ListCard.vue`:** Small, single-purpose components with typed props and emits. Well under the 250-line limit.
- **TypeScript strict mode:** `tsconfig.json` has `"strict": true`, `"noUnusedLocals": true`, and `"noUnusedParameters": true` all enabled. The project respects this configuration throughout.
- **`defineProps<{...}>()` and `defineEmits<{...}>()`:** Consistently used throughout all components reviewed. No runtime prop definitions found.
